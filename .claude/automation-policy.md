# 自動化ポリシー

このドキュメントは、Frontend / Design / CI/CD の全チームが共通で従う **作業自動化のルール** を定める。

## 1. 目的

各セッションの **開始時** と **終了時** に繰り返される定型作業を観察し、頻度が一定を超えたものをプロジェクト固有 Skill として自動化する。これにより、

- 手作業の漏れ（テスト忘れ・ビルド忘れ・コミット漏れ）を減らす
- セッション間で「何を毎回やるか」のばらつきを抑える
- 各チームの CLAUDE.md にプロジェクト固有の運用知が蓄積される

## 2. 自動化のトリガー

> **同一の作業 × 同一のタイミング（セッション開始 or 終了）が 3 回記録されたら、自動化対象**とする。

「同一の作業」とは、ツール呼び出し列が実質的に同じ操作（差分は引数の軽微な違い程度）を指す。  
「同一のタイミング」は **session start**（最初のユーザー発話の直後まで）/ **session end**（最後のユーザー発話の前後）の二択。

## 3. 観察と記録

セッション中に開始/終了で行ったタスクは、`.claude/memory/repeated-tasks.md` に追記する。フォーマット：

```
| タスク名 | チーム | タイミング | 日付 | コミット/ブランチ等の補足 |
|---------|--------|-----------|------|---------------------------|
| startup-test 実行 | frontend | start | 2026-05-06 | main |
```

- 1セッションにつき同一タスクは1行（重複記録しない）
- 担当チーム判定は §6 参照

## 4. 自動化フェーズ

カウントが 3 に達したら、次の手順を踏む：

1. **Skill 作成** — `.claude/skills/<skill-name>/SKILL.md` を作る。YAML frontmatter は最低限以下：
   ```yaml
   ---
   name: <skill-name>
   description: いつ使うか（when 句で書く）
   ---
   ```
   本体に「実行コマンド」「成功条件」「失敗時のフォールバック」を書く。スクリプトが必要なら同ディレクトリ配下に `scripts/` を切る。

2. **担当チームの CLAUDE.md に追記** — そのチームの `CLAUDE.md` 末尾に以下のセクションを追加（既にあれば追記）：
   ```markdown
   ## 自動化済み Skill
   - `<skill-name>` — <用途>。発動: セッション<開始|終了>。
   ```

3. **memory に完了印** — `.claude/memory/repeated-tasks.md` の該当行末に `→ automated: <skill-name>` を追記。同じタスクの再カウントは止める。

## 5. 運用フェーズ

以降、該当タイミングでは Claude Code の `Skill` ツールで `<skill-name>` を呼び出して実行する。手作業に戻さない（戻したくなった場合はポリシー違反として user に報告）。

## 6. 担当チーム判定

| 自動化したい作業 | 配置先 |
|------------------|--------|
| `npm test` / `vitest` / dev server 起動 / UI動作確認 | frontend/CLAUDE.md |
| デザイントークン整合チェック / Figma 同期 / CSS lint | design/CLAUDE.md |
| build / GitHub Actions / Pages デプロイ前検証 / PR生成 | cicd/CLAUDE.md |
| 横断的なもの（git status 確認、メモリ整理 等） | このポリシー文書のセクション末尾 |

## 7. ポリシー違反

- カウント 3 未満で自動化しない（先回りで Skill を量産しない）。
- 他チーム CLAUDE.md への追記は行わない（担当領域分離は CLAUDE.md の「禁止事項」で明示済）。
- Skill が陳腐化した場合は削除し、`repeated-tasks.md` の該当行に `→ retired: <理由>` を残す。
