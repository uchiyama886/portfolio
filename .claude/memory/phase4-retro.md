# Phase 4 振り返り — memory 整理・Skill 化

日付: 2026-05-07
担当チーム: 全チーム

## 学んだこと

### memory 管理
- `memory/README.md` のファイルリストは **新規 memory 追加時に同時更新**しないと漏れが生じる。今回 `vite-public-assets-base-url.md` が未記載で見落としていた。新ファイル作成 → README 更新を1セット作業として運用する。
- 各 memory ファイルの内容は 3 セッション後も陳腐化していなかった。**特定の技術決定（base URL, レスポンシブ戦略）** は長期間参照価値を保つ。一方でカウンタ的な情報（repeated-tasks.md）はセッションまたぎで状態更新が必須。

### Skill 化の判断基準
- `repeated-tasks.md` のログと自動化済みテーブルが **2つの別テーブル**になっているにも関わらず、`pr-open` が自動化済みテーブルに転記されていなかった。ログ側の `→ automated: <name>` だけでは不完全で、**テーブル転記まで行って初めて完了**とすべき。
- `lint+build` は 2/3 カウントで止まっていた。**Skill 化できるのに未着手の状態が長く続くと、手作業のばらつきが再発**する。Phase 4 で 3/3 にして即 Skill 化した。

## 失敗した点

- `repeated-tasks.md` の「自動化済み」テーブルと「ログ」の2カ所管理が同期漏れを生んだ。運用として「ログ側に → automated: を書いた時点でテーブルに同時転記」とルール明文化すべきだった。

## 次に活かす

- memory/README.md は新ファイル追加と同じコミットで更新する（1 コミットに両変更を含める）。
- repeated-tasks.md のカウントが 2/3 に到達したら、**その場で Skill 化する前提でメモ**を残し、次セッション開始時に必ず確認する。
- `dev server 起動 + 視覚レビュー疎通` は 1/3 のまま。次回 frontend チームがビジュアル実装をする際に 2/3 → 3/3 の見込み。
